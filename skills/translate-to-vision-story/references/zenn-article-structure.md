# Zenn Article Structure (3 段構成)

`translate-to-vision-story` skill が出力する Zenn 記事下書きの標準構造。

## 全体構成

```
[Title]: 1 行で「やったこと + ビジョンへの繋がり」を示す
[Frontmatter]: emoji / type / topics / published

[1. 背景] (200-400 字)
  - なぜこのプロジェクトを始めたか
  - 解決したい課題は何だったか
  - ビジョン要素 V_x との繋がりを 1 文で示す

[2. 取り組み] (本文の 60-70%)
  - やったことの技術的詳細
  - 重要な設計判断 (検討した代替案と却下理由を含む)
  - コード例 / 図 / コマンド例

[3. ビジョンへの繋がり] (200-400 字)
  - この活動がビジョン要素 V_x をどう前進させたか
  - 「自分軸の積み上げ」としての意味づけ
  - 次の活動 (次回記事の予告) があれば書く
```

## Frontmatter テンプレート

```yaml
---
title: "(Title)"
emoji: "💭"
type: "tech"  # or "idea"
topics: ["claudecode", "(domain)", "(tech)"]
published: false  # ドラフト状態
---
```

## 各段の書き方

### 1. 背景

- **悪い例**: 「○○について書きます。最近○○が流行っているので...」
- **良い例**: 「freeeサインの認証統合で、`devise` を捨てて freee 共通基盤に乗せる判断をした。これは『非同期ドリブン (V1)』を実装で示す試みだった。」

冒頭でビジョン整合点を明示することで、読者が「これは自分軸の活動だ」と認識できる。

### 2. 取り組み

- 設計判断には必ず「検討した代替案と却下理由」を書く (ADR-003 で確立した型)
- コード例は実際に使った言語 (Ruby/TypeScript/Markdown 等) で書く
- 図はテキストアスキー or mermaid を使う

### 3. ビジョンへの繋がり

- 「○○ができるようになった」だけで終わらせない
- 「この活動がビジョン要素 V_x の理解を深めた」「次は V_y に取り組む」と次の手を示す
- 「失敗した点 / ビジョンに整合しなかった点」も正直に書く (branding の真正性を高める)

## 段ごとの推定文字数

| 段 | 文字数 | 比率 |
|---|---|---|
| 背景 | 200-400 字 | 10-15% |
| 取り組み | 1500-3000 字 | 60-70% |
| ビジョンへの繋がり | 200-400 字 | 10-15% |
| (まとめ・参考リンク) | 100-300 字 | 5-10% |
| **合計** | 2000-4000 字 | 100% |

Zenn 記事の標準的な長さに収める。
