# 委譲実行 (qa-ui)

Task で委譲実行された場合の読み替えを定める。単独起動（ユーザーがメイン会話で直接起動）の現行動作は変えない。読み替えの発動条件は「AskUserQuestion が利用可能ツールに無いこと」（= 返答を待つことが構造的にできないこと）とし、subagent であること自体を理由にしない。

## 入力解決の優先順位 (プランファイルパス・ベース URL)

Step 2 のベース URL・Step 3 のプランファイルパスは次の順で解決する: ① 起動プロンプト本文の明示指定 → ② セッションコンテキスト（単独起動時のみ有効。委譲実行では引き継がれない）→ ③ 各 Step 本文が定めるファイル探索・プラン記載値の採用。①〜③ いずれでも解決しない場合、Step 2 / Step 3 の「ユーザーに尋ねる」を「不足入力: <項目名> を最終メッセージで返し、返答を待たずに終了する」に読み替える。

## Step 4 (人間委譲) の分割実行契約

Step 4 の「手順書を提示し、人間の返答を待って停止する」は、AskUserQuestion が使えない委譲実行では文字どおりに実行できない（subagent は最終メッセージを返して終了する構造のため）。次のとおり実行を分割する:

1. 委譲実行では手順書全文を最終メッセージとして返し、その時点で終了する（待たない）。
2. 呼び出し元は人間から回答を得たうえで本 skill を再起動する。台帳（`<プラン名>.qa-ledger.md`）が Step 3.5 以降の状態の正本であり、記帳は追記のみで「最新行が勝つ」規則を保つため、再起動したエージェントは台帳を Read するだけで各 QA-ID の状態を復元でき、Step 4 のラウンド判定を安全に再開できる。
3. 最終メッセージには手順書全文に加え、上記 1〜2（人間の回答を得たうえで台帳から再開する旨）を明記する。

この分割実行契約は Orchestrated モード宣言の有無に関わらず適用する。[orchestrated-mode.md](orchestrated-mode.md) が定める「Step 4 は escalation ledger の記帳対象外」は記帳の要否を定めるものであり、この分割実行契約とは独立に成立する。

## automation モードの Task 不可時

Step 4 automation サブセクションは `ui-evaluator` を Task で起動する。Task (Agent) ツールが利用可能ツール一覧に無い場合、ui-evaluator は ChromeDevTools MCP 経由の検証が前提のため代替手段が無い。この場合は automation を実行せず、対象 QA-ID を台帳に `検証不能(真の制約)` として記帳し（真の制約: 委譲実行文脈で Task 起動手段が無いこと）、他の QA-ID の処理は止めない。
