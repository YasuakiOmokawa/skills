# 実例: 実現性不明 → 捨て spike → 規約準拠で作り直し → 決定は ADR に保全

ある実プロジェクトの流れを**匿名化**して、code-first (spike → 作り直し) の 1 周がどう回るかを示す。製品名・リポジトリ・PR 番号・チケット ID・フラグ名・環境名などの固有情報は意図的に除いてある。再利用できるのは「規律」だけなので、それだけを残す。

## 状況

ある画面を、左にプレビュー・右に入力フォームを並べた新レイアウトに作り替えたい。PRD はあるが **技術実現性が不明**だった:

- 既存の保存エンドポイントを流用したまま新 UI を載せられるか
- プレビュー上に入力位置のオーバーレイを出せるか
- 左右ペインのフォーカス同期が成立するか

危険は「実現可能性・既存基盤の流用可否」= **可逆**(view 層に閉じ、捨てられる)。→ code-first を選択。

## 2 つの PR に分けた

### PR-A = 捨て spike (loop step 1)

- 機能フラグ (**既定 OFF**) で gate し、本番ユーザーに影響が出ない構成で新レイアウトを実機検証
- **検証用 throwaway と割り切り、責務分離 (層分割・生成クライアント化) はしない** — ロジックを 1 ファイルに集約。入力タイプも一部のみ対応 (残りは仮表示で代替)
- PR 説明に **「やらなかったこと」を列挙** — 本実装で潰す宿題が消えないようにした
- 結果: 実装可否と UX 仮説を実機で接地 → **この PR は CLOSED (コードは捨てる)**

### PR-B = 規約準拠で作り直し (loop step 2-3 → 4-6)

- spike が規約を無視して書かれていた (1 ファイル集約) ため、**リファクタでなく作り直し**を選択 — 可逆な危険だが「逸脱がひどくリファクタ費 > 作り直し費」という**コスト軸**の判断
- 既存コードベースの層構成に沿って設計し直し、既存のサーバ保存基盤をレスポンス形式ごと流用して **影響範囲を表示層に閉じた**
- **決定を ADR 数本に保全** (テーマのみ示す):
  - 新旧の並行稼働と出し分け (機能フラグ + 既定値 + 個別選好の独立した軸 / 障害時に既定へ戻せる評価順序)
  - 表示オーバーレイをフロント算出にする実現性判断 (画面表示と最終出力は別物・ピクセル一致は不要)
  - サーバ通信は既存基盤を再利用し新 API を作らない理由 + 移行完了後に開く選択肢
  - フロント構成の「読む地図」(import を辿るだけでは掴めないサイドチャネルの集約)
- 各 ADR に **却下した代替案**と「将来の読者が蒸し返しやすいので退けた理由を残す」を明記

## この実例から取り出せる規律

1. **spike は flag / 既定 OFF で本番非接触にする** — 捨て前提でも、本番に既成事実 (migration・公開エンドポイント・本番データ) を作らない
2. **spike の「やらなかったこと」を書き出す** — 本実装で潰す宿題が消えない
3. **規約逸脱がひどければリファクタでなく作り直し** — 可逆な危険でもこのコスト軸の分岐はある (「効かせる規律」§ の brownfield 版)
4. **捨てるのはコード、残すのは決定** — 実現性の学び (なぜ可能と判断したか・却下した代替案) を ADR に固める。spike を CLOSED しても知識が漏れない
5. **技術実現性は ADR、UX 仮説の検証結果は DD (設計文書) に分けて置く** — 技術 ADR に UX 検証を混ぜない

## loop との対応

| この実例 | loop step |
|---|---|
| PR-A (捨て spike) | step 1 (spike → 接地 verdict) |
| PR-B 本実装 | step 2 (Code-A) — ただし**リファクタでなく作り直し**で作った変種 |
| PR-B 層構成・配置の整え | step 3 (`/review-design`) |
| PR-B の ADR・設計文書 | step 5-6 (決定の保全・デリバリー化) |

この例は「`The loop` 表の step 2 を、Code-A のリファクタ (Code-A′) ではなく作り直しで起こした」ケース — `効かせる規律` と `Common mistakes` が言う「blast radius が大きい / 規約逸脱がひどい時だけ作り直し」に当たる。
