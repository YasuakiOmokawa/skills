## シナリオ: QA-ID カバレッジマトリクスに計画されたテストが diff に無い (未実装の検出)

`plan-diff-reviewer` として、確定プランファイルの QA-ID カバレッジマトリクスに以下の行がある:

| QA-ID | 出典 | カテゴリ | テストファイル | テストケース | 実行コマンド |
|-------|------|---------|----------------|--------------|--------------|
| QA-E-01 | 権限が無いユーザーが更新APIを叩くと403を返す | 異常系 | spec/requests/x_spec.rb | it "QA-E-01: ..." | `bundle exec rspec spec/requests/x_spec.rb -e "QA-E-01"` |
| QA-E-02 | 正常系: 権限があるユーザーが更新に成功する | 正常系 | spec/requests/x_spec.rb | it "QA-E-02: ..." | `bundle exec rspec spec/requests/x_spec.rb -e "QA-E-02"` |

対象 diff (`git diff develop..HEAD`) は `spec/requests/x_spec.rb` に `it "QA-E-02"` のテストケースを追加しているが、`it "QA-E-01"` に相当するテストケースは diff 中のどこにも存在しない。プロダクションコード側は権限チェックの分岐を実装済み。

### Requirements checklist

1. [critical] QA-E-01 を「未実装」として Critical で検出する（QA-ID カバレッジマトリクスに記載の auto QA-ID が対応するテストを持たない、の判定基準に該当）
2. [critical] QA-E-01 の指摘に、diff 中に `it "QA-E-01"` に相当する記述が無いこと（grep 相当の確認結果）を diff 根拠として明示する。プロダクションコードの実装有無だけで判定せず、テストの不在を独立に指摘する
3. QA-E-02 は diff に対応するテストケースが存在するため、誤って未実装・乖離ありとして検出しない（実装済み項目の過検出をしない）
4. 総合判定を「Critical あり」とし、他の乖離 (計画外差異等) が無ければ乖離一覧に QA-E-01 の 1 行のみを含める

## シナリオ: SKILL.md 委譲実行 — 確定プランパス・起点ブランチとも明示 (median)

todo アプリの git リポジトリが `feature/priority` ブランチで、確定プラン `plan-priority-final.md` に基づく優先度機能の実装を完了している。実装手順 5 は「優先度の異なる複数のタスクを追加し、`listTasks` の出力順が優先度の高い順になることを検証する回帰テストを追加する」と明記しているが、追加された自動テストは優先度未指定時のデフォルト値検証 1 件のみで、ソート順を検証するテストは無い。QA-ID カバレッジマトリクスは該当 AC を「対応する自動テストが未実装のため manual 扱い」と明記して事後的に manual 側へ割り当てている。

review-plan-diff を、確定プランファイルパス・起点ブランチ (`main`)・対象リポジトリの絶対パスをいずれも起動プロンプト本文で明示した状態で、委譲実行 (`AskUserQuestion` が利用可能ツール一覧に無い前提) として実行する。

### Requirements checklist

1. [critical] 実装手順 5 の回帰テスト未実装を Critical として検出し、報告に明記する (QA-ID カバレッジマトリクスが manual 扱いに付け替えていても、実装手順の記述義務は免除しない)
2. [critical] `plan-diff-reviewer` を nested 起動する際、`${CLAUDE_PLUGIN_ROOT}` を生文字列のまま埋め込まず、実際の絶対パスに解決したうえで渡す
3. `Task` (または委譲実行環境で提供される同等のサブエージェント起動ツール) で `plan-diff-reviewer` を起動する
4. 確定プランの分析ファイル (`<プラン名>.analysis.md`) の存在を確認し、ある場合は AC/MECE 分析結果との突き合わせに使う
5. 起点ブランチ `main` との diff (`git diff main..HEAD` 相当) を実際に取得して突き合わせる
6. 最終判定 (Critical あり) が乖離一覧とともに明確に提示されている

## シナリオ: SKILL.md 委譲実行 — 起点ブランチ未指定・preflight.md 欠落 (edge)

同じ todo アプリのフィクスチャを使うが、`<プラン名>.preflight.md` を意図的に置かず、起動プロンプト本文にも起点ブランチの指定を含めない。確定プラン本文の「ブランチ戦略」節には `feature/priority` (実装を継続する作業ブランチ) の記載があるが、これは diff の比較元となる起点ブランチのラベル付き記載ではない。

review-plan-diff を、確定プランファイルパスのみ明示し起点ブランチは指定しない委譲実行として実行する。

### Requirements checklist

1. [critical] 起点ブランチをプラン本文・preflight.md のいずれからも確定できない場合、「不足入力: 起点ブランチ」相当の内容を最終メッセージに含めてその場で終了する (返答を待って停止・ハングしない)
2. [critical] `<プラン名>.preflight.md` の存在を実際に確認したうえで不足と判断する (確認を省いて当てずっぽうで起点ブランチを確定しない)。プラン本文「ブランチ戦略」節の作業ブランチ記載を、起点ブランチのラベルが無いにもかかわらず起点ブランチとして誤採用しない
3. 不足入力の報告に具体的な項目名 (起点ブランチ) を明示する (曖昧な「情報不足」で終わらない)
4. 起点ブランチ以外の入力解決 (分析ファイルの有無確認等) は通常どおり進める
5. 起点ブランチ未確定のまま `Task` 相当のツールで `plan-diff-reviewer` を起動しない (diff 取得・レビュー起動を試みる前に停止する)
