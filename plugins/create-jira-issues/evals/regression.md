# regression eval (empirical-prompt-tuning 収束時保存)

収束記録: 2026-07-11 (map-user-stories タスク TSV 新契約への同期)。マッピング表を専用 3 列 (やること/やらないこと/完了条件) の直読みに更新後、fresh executor による契約パース机上シナリオを 4 回実行し全 [critical] ○。分解規則の precision 残差 (そのまま転記と Decision Table の関係 / 活用形調整の許容 / 30 文字以下×複数動詞の優先 / 明示記号なしの分解境界 / 助詞調整 / 依存列 `-` の空欄同義) を都度明文化し、4 回目で分解・転記規則の迷い 0 を確認して収束。
用途: **regression 検出器** (capability 改善の信号としては使わない)。本 skill を変更する PR では
fresh executor (blank slate, Task dispatch) で下記シナリオを再実行し、全 [critical] ○ を確認してから merge する。
実行方法は empirical-prompt-tuning の「Subagent invocation contract」に従う (成果物はインライン、ファイル編集禁止)。

## シナリオ: 9 列タスク TSV (30 文字超・明示記号なしの複数動詞を含む) の契約パースと description 転記

対象プランファイルの `## タスクリスト` 直下に、Step 1 のパース契約どおり US_ID/Task_ID/タスク名/やること/やらないこと/完了条件/依存タスク/Jira/備考 の 9 列 TSV が渡される:

```tsv
US_ID	Task_ID	タスク名	やること	やらないこと	完了条件	依存タスク	Jira	備考
US-001	T-009	認可ミドルウェアの追加	既存 middleware chain に認可チェックを追加する	新規ロールの追加	spec/requests/auth_spec.rb が green	-	XPROJ-508	
US-001	T-010	APIレスポンスへのエラーコード付与	認可チェックミドルウェアを追加してAPIレスポンスに403エラーコードを付与する	権限ロールの新規追加	未認可アクセス時に403エラーコードを返すテストが green	T-009		
```

T-010 の「やること」列は 30 文字超・複数動詞で、区切り記号「+」「、」を含まない (て形の動詞境界のみ)。依存タスク列は T-009 (Jira 済み) を指す。

### Requirements checklist
1. [critical] 9 列 TSV を Step 1 のパース契約で正しく読み取り、列不足・列順違いと誤判定してパースを停止しない
2. [critical] サブタスク description の「やること」セクションを「やること」列の本文からのみ転記する (「タスク名」列や他列の内容を合成しない)
3. T-010 の「やること」列 (30 文字超・複数動詞・明示記号なし) を、接続助詞 (て形) の動詞境界「追加して」で「認可チェックミドルウェアを追加する」「APIレスポンスに403エラーコードを付与する」の2項目に分解する (30文字以下ルールを誤って適用し1項目のままにしない)
4. T-010 の着手条件を、依存タスク列 T-009 に対応する Jira キー (XPROJ-508) へ展開する (Task_ID をそのまま転記しない)
5. summary はタスク名列をそのまま採用する (変形・要約をしない)
6. 「やらないこと」「完了条件」列もそれぞれ専用列からそのまま転記され、US 側専用のプレフィックス剥離処理 (`やらない:` 等) を誤ってサブタスクに適用しない
