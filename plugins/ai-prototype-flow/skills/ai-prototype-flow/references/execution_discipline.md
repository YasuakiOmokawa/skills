> **実行規律の詳細**: /ai-prototype-flow の Workflow で `/xxx` チェーン / 停止点 / 非該当判定 / 再入イベントの取り扱いを定める。フェーズのプロンプト列を実行する際に本ファイルを Read すること。

## 実行規律

- 指定フェーズのプロンプト列を上から順に実行する。プロンプト列に現れる `/xxx` は、
  行頭か文中かを問わず Skill ツールで起動する。`→` で連結された複数 skill は
  1 つの作業項目として順に起動し、間に停止点を置かない。行内に地の文の指示が
  続く場合 (例: 末尾の「〜と照合」) も同じ作業項目の一部として続けて実施する。
  チェーン内の一部要素だけが実施不能な場合 (未インストール・ルールファイル
  なし等) は、その要素だけをスキップ記録し、残りの要素は続けて実施する
  (チェーン全体を止めない)
- `(人間: ...)` 行は停止点。依頼内容を提示してユーザーの返答を待つ
- `! <command>` 行はユーザーが打つ前提の記法で、停止点。コマンドを提示して実行結果の
  共有を待つ (後続ステップが結果を入力に使うため。例: react-doctor の指摘を
  /polish-before-commit が集約する)。代行実行の明示指示があるときだけ Bash で実行して先へ進む
- 上記 3 記法 (`/xxx` / `(人間: ...)` / `! <command>`) のいずれでもない行
  (地の文の指示、ルールファイル照合が連結された行など) は、通常の作業指示として
  上から順に実施する。新しい記法の意味を発明しない
- 「停止点ではなく常設ルール」と注記された行は、その場では実行せず、フェーズ内で
  条件が成立した時だけ発火する待機ルールとして扱う
- 実装対象リポジトリが「なし」の案件 (文書のみで完結する案件) では、リポジトリの
  存在を前提とする作業項目・句を「非該当: リポジトリなし」と記録して飛ばす。
  判定基準は「リポジトリが無いと実施不能か」で、ブランチ作成・コード実装・
  コード品質パス・PR 作成は例示 (網羅列挙ではない — 「既存コードベースの慣習に
  従う」のような修飾句も基準に該当すれば飛ばす。コード・コミット・PR を対象と
  する skill や句は、コード自体が存在しないため常に基準に該当する。文書 —
  プラン・md — を入力に取りうる skill (/review-plan-diff や md 文書照合の
  prose 系等) は、案件に実在する文書成果物があれば適用対象で、非該当にしない)。
  この非該当判定は 3 記法 (`/xxx`・`(人間: ...)`・`! <command>`) の扱いに
  先立つ — 非該当の行は停止点にもならない。判定は出現行ごとに行う —
  同じ skill 名が複数行に現れたら行ごとに用途で再判定する。行内注記
  (「常に実行」等) はその注記が述べる条件との関係だけを定めるもので、
  この非該当判定を上書きしない。複数の非該当根拠が同時に成立する行は、
  リポジトリなし等の構造的な判定を主根拠として記録する (他の根拠の併記は任意)。
  リポジトリ前提の句と文書作業の
  句が同一行に混在する場合は、行を句ごとに分割して判定結果を並記し、文書作業の
  句は実施する (例: 「本実装を設計。既存コードベースの慣習に従うよう設計して
  /grill-with-docs」→ 設計と /grill-with-docs は実施、慣習句のみ非該当)。
  逐次動作の複合文も同様に句で割る (例: 「/iterate-with-prototypes に従い仮説を
  ledger にしてから spike して」→ skill 起動と ledger 化は先行句として実施、
  spike 句のみ非該当)。
  progress-ledger のスライス状態 (5 値) も同じ基準で縮退させる — 非該当工程に
  対応する状態 (QA中・PR済 等) は経由・記帳せず、実施した最後の該当工程の状態へ
  直接更新する (文書のみのスライスは実装完了 = 完了)
- 起動しようとした `/xxx` が見つからない場合、まず類似名の skill が無いか確認し、
  あれば読み替えてよいかユーザーに確認する。類似名も無ければ
  「スキップ: <skill 名> 未インストール」と記録して先へ進む
- `再入 <イベント>` 指定時は、正本の「再入ルール」節から該当項目を選ぶ。戻り先フェーズが
  定義されている項目はそのフェーズの手順として実行し、戻り先を持たない運用ルール項目
  (正本管理等) は該当ファイルへ適用して終了する (Step Z は実行しない — 記帳は
  正本の再入ログ共通規則のみで、案件プランファイルの進捗欄には書かない)。開始時に progress-ledger の
  `## 再入ログ` 行数と案件ディレクトリの差分設計ファイル (`dd_prd_delta*.md`) 数を
  突き合わせ、食い違いは「記帳漏れ疑い」としてユーザーに提示する
  (単一の記帳経路に頼らない。ログ 0 件でもファイルが再入の実績を語る)。
  検出・提示までが必須動作で、過去分の遡及記帳は行わずユーザー確認に委ねる
  (不確実な過去を断定して記帳しない)。今回の再入処理自体は継続してよい
