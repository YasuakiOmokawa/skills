# dispatcher モード別手順 (案件初期化 / status)

SKILL.md Step 0 手順 2 で **案件初期化** または **status 質問** と判定したときに読む詳細手順。
どちらもフェーズを実行しない分岐で、common path (フェーズ実行) とは相互排他。
本文中の「Step 0」「用語の既定」「実行規律」は SKILL.md の同名節を指す。

## 案件初期化 (gdocs URL 起点)

1. URL から doc ID を抽出する (`/document/d/<docID>/` の部分)。`?tab=t.<tabID>` が
   あればタブ ID も控える (スナップショット範囲は指定タブのみになる)。
   `#heading=...` フラグメントは無視する — 取得単位は doc かタブで、見出し単位の
   エクスポートは存在しない
2. 案件名を確定する: ユーザーの明示指定があればそれ。無ければ先に scratchpad へ
   エクスポートだけ実行し (正本手順 2 と同じコマンド — deny 時の縮退列も同手順に
   従う。全文へ縮退したときはタブ先頭見出しでなく doc タイトル = 出力ファイル名で
   命名する)、doc タイトル (= 出力
   ファイル名。タブ指定時は出力の先頭見出し) から snake_case の短い英名を導出
   して宣言する。導出元は組織定型の接頭辞 (`Brief／PRD_` 等) を除いた実質名と
   し、doc 本文に「機能名」等の自己申告欄があればその値を優先する。タイトル取得の
   ためだけに二度エクスポートしない — scratchpad の生データは手順 4 でそのまま使う
3. `~/.claude/ai-prototype-flow-projects/*/plan_*.md` の `PRD gdocs:` 行を grep し、
   同一 doc ID (タブ違い含む) の既存案件があれば「別スコープの新規案件か、
   既存案件への再入か」を確認する
   (確認できない委譲実行等では、既存 1 件までなら新規作成し重複の旨を件数・
   対象案件パス付きで報告に残す。既存 2 件以上なら重複の増殖を疑い、新規作成
   せず停止して確認する)。その上で
   `~/.claude/ai-prototype-flow-projects/<案件名>/` を作成する (プランファイルの
   作成と `PRD gdocs:` 行の記録は正本手順 1 が行う — ここで重複して書かない)
4. 正本の「案件初期化: PRD 凍結スナップショット」節の手順を実行する
   (手順 2 で scratchpad に生データを取得済みなら、エクスポートは案件
   ディレクトリへの mv で代替する)
5. プランファイルの進捗欄に `- <日付> 初期凍結スナップショット取得: <ファイル名>` を
   1 行追記し、保存パス一覧と gdocs 側 named version の依頼 (人間作業) を提示して
   終了する (フェーズは実行しない)

## status モード (現在地レポート)

現在地を尋ねる引数 (「いまどこ」「<案件名> っていまどのフェーズだっけ」等) で起動する。
フェーズは実行せず、記帳もしない。

1. Step 0 手順 3 と同じ手順で対象案件を特定する。ただし候補が複数でも選択質問は
   しない (相乗りできる確認質問がこのモードには無い) — 全候補の現在地を 1 つの
   レポートに並べて提示する
2. プランファイルと progress-ledger (あれば) を読み、次を 1 つのレポートで提示する
   (進捗欄を持たない候補は、記録形式外である旨と推定であることを明記する):
   案件名と案件ディレクトリ / 完了済みフェーズの全列挙 (各行とも進捗欄の日付と
   要点をそのまま転記) / 着手済み未完了フェーズ (中断) の有無 / 次の一手 (正本上の
   次フェーズ名。フェーズ3 進行中なら残スライスの状態別集計)
3. 提示したら終了する。続けてユーザーが実行を指示したら通常の実行に移る —
   現在地は提示済みなので Step 0 手順 2 の確認は繰り返さず、手順 4 (不足入力の
   確認) から続行する
