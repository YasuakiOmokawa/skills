# ai-prototype-flow

AI プロトタイプ駆動開発フローの **1 フェーズだけ**を実行する薄い dispatcher。
フローの正本は本 SKILL.md と同じディレクトリ配下の次のファイルで、本 skill は手順を複製しない (毎回 Read する):

```
references/ai_prototype_flow.md
```

正本が参照する DD テンプレート・実例 (組織固有のため同梱していない) は、正本の
各参照箇所が配置パスと未配置時の扱いを記す。このほかのスキル外依存は
`~/.claude/rules/japanese-writing.md` のみで、無ければ該当の照合句を
「スキップ: ルールファイルなし」と記録して飛ばす。

## 入力の確定 (Step 0)

1. 正本ファイルを Read する。無ければ停止してパスをユーザーに確認する
2. 引数からモードを特定する: `フェーズ0` / `1` / `2` / `2.5` / `3` / `再入 <イベント>` /
   `案件初期化 <gdocs URL>`。案件初期化 はフェーズを実行しない。正準形は
   `案件初期化 <URL>` だが、gdocs URL + スナップショット取得の指示文
   (例: `prd <URL> 初期凍結スナップショット保存して`) も同じ扱いにする。
   現在地を尋ねる文言 (「いまどこ」「どのフェーズだっけ」「進捗どうなってる」等) を
   含み、実行を指示する語 (「やって」「進めて」「実行」等) を含まない引数は
   status モードで処理する — フェーズを実行しない。
   **案件初期化・status のいずれかと判定したら `references/dispatcher_modes.md` を
   Read し、その節の手順に従う** (どちらもフェーズを実行しない分岐。詳細手順は
   common path のノイズを避けるため本 SKILL.md に複製しない)。
   引数なし・フェーズ語のない実行指示 (「続きやって」「次進めて」等) は無指定として
   扱い、案件のプランファイル・progress-ledger から現在地を推定して
   「フェーズ N からでよいか」を 1 回だけ確認する。進捗欄に着手行 (手順 6) があり
   対応する完了行が無いフェーズは中断とみなし、確認文言を
   「フェーズ N の続きからでよいか」にする。着手行を持たない完了行だけの履歴
   (手順 6 導入前の旧形式) は正常な記録として完了行のみで推定する。複数フェーズの明示指定
   (例: `フェーズ0と1`) は、各フェーズの末尾で Step Z を実施しながら指定範囲を
   連続実行する (ユーザーの明示指定は「自動連鎖」に当たらない)
3. 対象案件を特定する: 引数・会話に案件の指定 (案件名・パス・直近で扱った案件への
   言及) があればそれ。指定がパスでなく案件名だけのときは、下記 (a)(b) の探索を
   その名前で絞って案件ディレクトリに解決する。指定が無ければ (a) カレント
   ディレクトリ配下 (再帰探索 — 深さを絞らない) の `plan_*.md` /
   `*.progress-ledger.md` と (b) `~/.claude/ai-prototype-flow-projects/`
   配下の**両方**から候補を集める (fallback ではない — (a) に候補があっても
   (b) も見る。(a) が (b) を含む環境では重複を除く)。旧配置 (本 SKILL.md 配下の
   `projects/`) に案件ディレクトリが残っていた場合は
   `~/.claude/ai-prototype-flow-projects/` への移動を提案する (npx skills の
   update でスキルディレクトリ内は消えるため)。候補 1 件はそのまま採用して宣言し、複数件は
   候補一覧を手順 2 の現在地確認と同じ 1 回の質問に相乗りさせて選んでもらう
   (既定候補には進捗記録の更新が最も新しい案件を推す。比較は軽量に —
   フェーズ単位の進捗欄があればその最終日付、無ければ案件ディレクトリ内
   ファイルの最終更新時刻で代替し、候補の中身は深読みしない)。0 件は新規案件とみなし、
   「案件初期化 (gdocs 正本) かフェーズ0 開始か」を同じ質問で提案する
4. 対象案件の入力 (PRD / プランファイル / DD のパス。フェーズ2〜3 は実装対象
   リポジトリのパスとブランチも — 「既存コードベースの慣習に従う」設計・実装は
   これが無いと実行できない。再入時は差分の参照元 — 旧/新 Figma や PRD 版 — と
   取り込み先ブランチも含む) が会話・引数・案件ディレクトリ内の記録に無ければ、
   不足分をまとめて 1 回で聞く。聞くのは**ユーザーにしか答えられない値**
   (パス・参照先・GO/NOGO) だけ。自明なフォールバックで補える値 (新規プラン
   ファイルの作成先、実リポジトリ内案件の実装対象 = カレントリポジトリ等) は
   自己判断で補い、採用したフォールバックを宣言する
5. `フェーズ1` 指定時、フェーズ0 の成果物 (技術調査メモ) が会話・案件ディレクトリに
   見当たらなければ「フェーズ0 から始めるか、調査不要としてフェーズ1 に直行するか」を
   1 回確認する (省略してよいかはこの回答でユーザーが判断する。無言でフェーズ0 を飛ばさない)
6. フェーズ本体に入る直前に着手を記帳する: フェーズ0〜2.5 は案件プランファイルの
   進捗欄 (「用語の既定」参照) に `- <日付> フェーズ<N> 着手` を 1 行追記、
   フェーズ3 は progress-ledger の該当
   スライス行を「実装中」へ更新する (案件プランファイルにも簡易プランにも着手行は
   書かない — 記帳先を分散させない)。複数フェーズの連続実行では各フェーズの開始時に記帳する。
   中断からの継続では既存の着手行をそのまま起点にし、重複追記しない。
   セッションがフェーズ途中で切れても着手行が残るため、次回の現在地推定が未着手と
   中断を区別できる。status モード・案件初期化では記帳しない

用語の既定: **案件ディレクトリ** = 対象 PRD / プランファイルの親ディレクトリ。
gdocs URL 起点で新規に始める案件は、`~/.claude/ai-prototype-flow-projects/<案件名>/` を
作って案件ディレクトリとする (スキルディレクトリ内に作らない — npx skills の
update で消えるため)。
新規プランファイルの既定名 = PRD ファイル名の `prd_` を `plan_` に置換したもの
(PRD が無い場合は `plan_<案件名>.md`)。フェーズ3 の軽微スライス用
簡易プランの既定名 = `plan_<案件名>_slice<番号>.md` (番号を持たないスライスは
スライス名の snake_case を使う)。専用プランファイルを持たない案件では、
進捗の追記先は案件の主文書 (設計書等) の末尾とする。
**進捗欄** = プランファイルの `## 進捗` 節。進捗行 (着手・完了・スナップショット
取得) の追記先は常にこの節の末尾で、節が無ければ作る。ledger 等の他セクションが
後続していてもファイル物理末尾には書かない (進捗が 2 箇所に分裂すると
手順 2 の中断判定が読めなくなる)。
progress-ledger の既定名 = `<prd ファイル名>.progress-ledger.md`。ローカル prd
ファイルを持たない案件 (gdocs 正本等) はプランファイル名を代用する
(`plan_<案件名>.progress-ledger.md`)。凍結スナップショット
(`prd_gdoc_snapshot_*.md`) はローカル prd ファイルとして扱わない。

## 案件初期化 / status モード

いずれもフェーズを実行しない分岐。Step 0 手順 2 で判定したら
`references/dispatcher_modes.md` を Read し、その節の手順に従う
(案件初期化 = gdocs URL 起点の凍結スナップショット取得、status = 現在地レポート)。

## 実行規律

指定フェーズのプロンプト列を上から順に実行する。プロンプト列に現れる `/xxx` は
Skill ツールで起動、`(人間: ...)` は停止点、`! <command>` はユーザー実行前提の
停止点。詳細ルール (`→` 連結・チェーン内スキップ・非該当判定の縮退・再入イベント
の取り扱い) は `references/execution_discipline.md` を参照。フェーズのプロンプト列
に入る**直前**に本ファイルを 1 度 Read すること。

## フェーズ完了時 (Step Z)

各フェーズのプロンプト列の**末尾要素の完了**が Step Z の必須起動条件。末尾要素は
フェーズ0 = 技術調査メモ整理完了、フェーズ1/2 = /create-pr (draft) 完了、
フェーズ2.5 = /create-jira-issues or /map-user-stories 完了 or DD LGTM 受領、
フェーズ3 = /create-pr (ready for review) 完了。末尾要素の完了を検知したら
**次タスク (cron 監視 / merge / follow-up) に流れる前に**直ちに `references/step_z.md`
の手順 1-2 を実行する。フォローアップの自然な流れで Step Z を素通しするのは違反。

## 併用推奨 skill

- `/iterate-with-prototypes` — フェーズ1 の本体。仮説 ledger と spike の規律
- `/review-plan-diff` — フェーズ3 の実装後突き合わせ
- `/map-user-stories` — フェーズ2.5 のタスク分解
