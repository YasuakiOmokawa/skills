# ai-prototype-flow

AI プロトタイプ駆動開発フローの **1 フェーズだけ**を実行する薄い dispatcher。
フローの正本は本 SKILL.md と同じディレクトリ配下の次のファイルで、本 skill は手順を複製しない (毎回 Read する):

```
references/ai_prototype_flow.md
```

正本が参照する DD テンプレート・実例は組織固有の内容を含むため同梱していない。
リポジトリ README「設定値の生成」のセットアップスクリプト (`scripts/setup.sh`) が、
自組織の文書を `~/.claude/skills-config/ai-prototype-flow/` に配置する:
`dd_template.md` (DD テンプレート) / `dd_reference.md` (完成 DD の参考実例) /
`dd_prd_delta.md` (再入時の差分設計実例)。未配置のファイルは、参照する手順の側で
「スキップ: 未配置」と宣言して進める。
このほかのスキル外依存は `~/.claude/rules/japanese-writing.md` のみで、無ければ
該当の照合句を「スキップ: ルールファイルなし」と記録して飛ばす。

## 入力の確定 (Step 0)

1. 正本ファイルを Read する。無ければ停止してパスをユーザーに確認する
2. 引数からモードを特定する: `フェーズ0` / `1` / `2` / `2.5` / `3` / `再入 <イベント>` /
   `案件初期化 <gdocs URL>`。案件初期化 はフェーズを実行しない。正準形は
   `案件初期化 <URL>` だが、gdocs URL + スナップショット取得の指示文
   (例: `prd <URL> 初期凍結スナップショット保存して`) も同じ扱いにする。
   現在地を尋ねる文言 (「いまどこ」「どのフェーズだっけ」「進捗どうなってる」等) を
   含み、実行を指示する語 (「やって」「進めて」「実行」等) を含まない引数は
   status モードで処理する — フェーズを実行しない。
   **案件初期化・status のいずれかと判定したら `references/dispatcher_modes.md` を
   Read し、その節の手順に従う** (どちらもフェーズを実行しない分岐。詳細手順は
   common path のノイズを避けるため本 SKILL.md に複製しない)。
   引数なし・フェーズ語のない実行指示 (「続きやって」「次進めて」等) は無指定として
   扱い、案件のプランファイル・progress-ledger から現在地を推定して
   「フェーズ N からでよいか」を 1 回だけ確認する。進捗欄に着手行 (手順 6) があり
   対応する完了行が無いフェーズは中断とみなし、確認文言を
   「フェーズ N の続きからでよいか」にする。着手行を持たない完了行だけの履歴
   (手順 6 導入前の旧形式) は正常な記録として完了行のみで推定する。複数フェーズの明示指定
   (例: `フェーズ0と1`) は、各フェーズの末尾で Step Z を実施しながら指定範囲を
   連続実行する (ユーザーの明示指定は「自動連鎖」に当たらない)
3. 対象案件を特定する: 引数・会話に案件の指定 (案件名・パス・直近で扱った案件への
   言及) があればそれ。指定がパスでなく案件名だけのときは、下記 (a)(b) の探索を
   その名前で絞って案件ディレクトリに解決する。指定が無ければ (a) カレント
   ディレクトリ配下 (再帰探索 — 深さを絞らない) の `plan_*.md` /
   `*.progress-ledger.md` と (b) 本 SKILL.md 配下の
   `projects/` の**両方**から候補を集める (fallback ではない — (a) に候補があっても
   (b) も見る。(a) が (b) を含む環境では重複を除く)。候補 1 件はそのまま採用して宣言し、複数件は
   候補一覧を手順 2 の現在地確認と同じ 1 回の質問に相乗りさせて選んでもらう
   (既定候補には進捗記録の更新が最も新しい案件を推す。比較は軽量に —
   フェーズ単位の進捗欄があればその最終日付、無ければ案件ディレクトリ内
   ファイルの最終更新時刻で代替し、候補の中身は深読みしない)。0 件は新規案件とみなし、
   「案件初期化 (gdocs 正本) かフェーズ0 開始か」を同じ質問で提案する
4. 対象案件の入力 (PRD / プランファイル / DD のパス。フェーズ2〜3 は実装対象
   リポジトリのパスとブランチも — 「既存コードベースの慣習に従う」設計・実装は
   これが無いと実行できない。再入時は差分の参照元 — 旧/新 Figma や PRD 版 — と
   取り込み先ブランチも含む) が会話・引数・案件ディレクトリ内の記録に無ければ、
   不足分をまとめて 1 回で聞く。聞くのは**ユーザーにしか答えられない値**
   (パス・参照先・GO/NOGO) だけ。自明なフォールバックで補える値 (新規プラン
   ファイルの作成先、実リポジトリ内案件の実装対象 = カレントリポジトリ等) は
   自己判断で補い、採用したフォールバックを宣言する
5. `フェーズ1` 指定時、フェーズ0 の成果物 (技術調査メモ) が会話・案件ディレクトリに
   見当たらなければ「フェーズ0 から始めるか、調査不要としてフェーズ1 に直行するか」を
   1 回確認する (省略してよいかはこの回答でユーザーが判断する。無言でフェーズ0 を飛ばさない)
6. フェーズ本体に入る直前に着手を記帳する: フェーズ0〜2.5 は案件プランファイルの
   進捗欄 (「用語の既定」参照) に `- <日付> フェーズ<N> 着手` を 1 行追記、
   フェーズ3 は progress-ledger の該当
   スライス行を「実装中」へ更新する (案件プランファイルにも簡易プランにも着手行は
   書かない — 記帳先を分散させない)。複数フェーズの連続実行では各フェーズの開始時に記帳する。
   中断からの継続では既存の着手行をそのまま起点にし、重複追記しない。
   セッションがフェーズ途中で切れても着手行が残るため、次回の現在地推定が未着手と
   中断を区別できる。status モード・案件初期化では記帳しない

用語の既定: **案件ディレクトリ** = 対象 PRD / プランファイルの親ディレクトリ。
gdocs URL 起点で新規に始める案件は、本 SKILL.md と同じディレクトリの
`projects/<案件名>/` を作って案件ディレクトリとする。
新規プランファイルの既定名 = PRD ファイル名の `prd_` を `plan_` に置換したもの
(PRD が無い場合は `plan_<案件名>.md`)。フェーズ3 の軽微スライス用
簡易プランの既定名 = `plan_<案件名>_slice<番号>.md` (番号を持たないスライスは
スライス名の snake_case を使う)。専用プランファイルを持たない案件では、
進捗の追記先は案件の主文書 (設計書等) の末尾とする。
**進捗欄** = プランファイルの `## 進捗` 節。進捗行 (着手・完了・スナップショット
取得) の追記先は常にこの節の末尾で、節が無ければ作る。ledger 等の他セクションが
後続していてもファイル物理末尾には書かない (進捗が 2 箇所に分裂すると
手順 2 の中断判定が読めなくなる)。
progress-ledger の既定名 = `<prd ファイル名>.progress-ledger.md`。ローカル prd
ファイルを持たない案件 (gdocs 正本等) はプランファイル名を代用する
(`plan_<案件名>.progress-ledger.md`)。凍結スナップショット
(`prd_gdoc_snapshot_*.md`) はローカル prd ファイルとして扱わない。

## 案件初期化 / status モード

いずれもフェーズを実行しない分岐。Step 0 手順 2 で判定したら
`references/dispatcher_modes.md` を Read し、その節の手順に従う
(案件初期化 = gdocs URL 起点の凍結スナップショット取得、status = 現在地レポート)。

## 実行規律

- 指定フェーズのプロンプト列を上から順に実行する。プロンプト列に現れる `/xxx` は、
  行頭か文中かを問わず Skill ツールで起動する。`→` で連結された複数 skill は
  1 つの作業項目として順に起動し、間に停止点を置かない。行内に地の文の指示が
  続く場合 (例: 末尾の「〜と照合」) も同じ作業項目の一部として続けて実施する。
  チェーン内の一部要素だけが実施不能な場合 (未インストール・ルールファイル
  なし等) は、その要素だけをスキップ記録し、残りの要素は続けて実施する
  (チェーン全体を止めない)
- `(人間: ...)` 行は停止点。依頼内容を提示してユーザーの返答を待つ
- `! <command>` 行はユーザーが打つ前提の記法で、停止点。コマンドを提示して実行結果の
  共有を待つ (後続ステップが結果を入力に使うため。例: react-doctor の指摘を
  /polish-before-commit が集約する)。代行実行の明示指示があるときだけ Bash で実行して先へ進む
- 上記 3 記法 (`/xxx` / `(人間: ...)` / `! <command>`) のいずれでもない行
  (地の文の指示、ルールファイル照合が連結された行など) は、通常の作業指示として
  上から順に実施する。新しい記法の意味を発明しない
- 「停止点ではなく常設ルール」と注記された行は、その場では実行せず、フェーズ内で
  条件が成立した時だけ発火する待機ルールとして扱う
- 実装対象リポジトリが「なし」の案件 (文書のみで完結する案件) では、リポジトリの
  存在を前提とする作業項目・句を「非該当: リポジトリなし」と記録して飛ばす。
  判定基準は「リポジトリが無いと実施不能か」で、ブランチ作成・コード実装・
  コード品質パス・PR 作成は例示 (網羅列挙ではない — 「既存コードベースの慣習に
  従う」のような修飾句も基準に該当すれば飛ばす。コード・コミット・PR を対象と
  する skill や句は、コード自体が存在しないため常に基準に該当する)。
  この非該当判定は 3 記法 (`/xxx`・`(人間: ...)`・`! <command>`) の扱いに
  先立つ — 非該当の行は停止点にもならない。判定は出現行ごとに行う —
  同じ skill 名が複数行に現れたら行ごとに用途で再判定する。行内注記
  (「常に実行」等) はその注記が述べる条件との関係だけを定めるもので、
  この非該当判定を上書きしない。複数の非該当根拠が同時に成立する行は、
  リポジトリなし等の構造的な判定を主根拠として記録する (他の根拠の併記は任意)。
  リポジトリ前提の句と文書作業の
  句が同一行に混在する場合は、行を句ごとに分割して判定結果を並記し、文書作業の
  句は実施する (例: 「本実装を設計。既存コードベースの慣習に従うよう設計して
  /grill-with-docs」→ 設計と /grill-with-docs は実施、慣習句のみ非該当)。
  逐次動作の複合文も同様に句で割る (例: 「/iterate-with-prototypes に従い仮説を
  ledger にしてから spike して」→ skill 起動と ledger 化は先行句として実施、
  spike 句のみ非該当)。
  progress-ledger のスライス状態 (5 値) も同じ基準で縮退させる — 非該当工程に
  対応する状態 (QA中・PR済 等) は経由・記帳せず、実施した最後の該当工程の状態へ
  直接更新する (文書のみのスライスは実装完了 = 完了)
- 起動しようとした `/xxx` が見つからない場合、まず類似名の skill が無いか確認し、
  あれば読み替えてよいかユーザーに確認する。類似名も無ければ
  「スキップ: <skill 名> 未インストール」と記録して先へ進む
- `再入 <イベント>` 指定時は、正本の「再入ルール」節から該当項目を選ぶ。戻り先フェーズが
  定義されている項目はそのフェーズの手順として実行し、戻り先を持たない運用ルール項目
  (正本管理等) は該当ファイルへ適用して終了する (Step Z は実行しない — 記帳は
  正本の再入ログ共通規則のみで、案件プランファイルの進捗欄には書かない)。開始時に progress-ledger の
  `## 再入ログ` 行数と案件ディレクトリの差分設計ファイル (`dd_prd_delta*.md`) 数を
  突き合わせ、食い違いは「記帳漏れ疑い」としてユーザーに提示する
  (単一の記帳経路に頼らない。ログ 0 件でもファイルが再入の実績を語る)。
  検出・提示までが必須動作で、過去分の遡及記帳は行わずユーザー確認に委ねる
  (不確実な過去を断定して記帳しない)。今回の再入処理自体は継続してよい

## フェーズ完了時 (Step Z)

フェーズ完了 = 非該当を除いた適用対象の作業項目が全て済んだ状態
(非該当の多寡は完了記帳を妨げない)。フェーズ3 は正本の「タスクごとに 1 周まわす」の
1 周 (= 1 スライス) を本節の適用単位とする — スライスごとに Step Z を実施して
停止し、次スライスへ自動で進まない。

1. 実行結果の要約と、正本に書かれた次フェーズ名を提示する。次フェーズを持たない
   フェーズ3 では、スライス 1 周の Step Z ごとに progress-ledger の残スライス
   (未着手・実装中) の集計を代わりに提示する。残スライスが 0 件なら「全スライス完了 —
   フロー終端」と提示する。存在しない次フェーズを作らない
2. 進捗を 1 行追記する: フェーズ0〜2.5 は案件プランファイルの進捗欄
   (「用語の既定」参照) に `- <日付> フェーズ<N> 完了: <要点 1 句>` 形式で追記
   (Step 0 手順 6 の着手行は消さない — 着手→完了の対がフェーズの履歴になる)、
   フェーズ3 は
   `<prd>.progress-ledger.md` の該当スライス行の状態を更新する (新規行を乱立させない)。
   再入実行時は、この通常のフェーズ完了行に**加えて** progress-ledger (命名は
   「用語の既定」参照) の `## 再入ログ` 節へ正本の形式で 1 行記帳する — 両方必須
   (台帳・節が無ければ作る)。この記帳を確認するまで Step Z を完了としない
3. **ここで終了する。次フェーズへ自動で進まない** (フェーズ間はユーザーが結果を見て
   次を指示する。オーケストレータによる自動連鎖は deliver-from-spec 廃止で
   利用者が退けた運用であり、本 skill で再発明しない)。例外は Step 0-2 の
   複数フェーズ明示指定のみ — その場合は指定範囲の最終フェーズの Step Z でのみ
   終了し、中間フェーズの Step Z では停止しない
