- executor への制約: (1) 対象 2 ファイルを編集しない (2) 恒久成果物は対象案件ディレクトリのみ
  (一時ファイルは scratchpad 可) (3) `(人間: ...)` 停止点・ユーザー確認は提示扱いで進行
  (4) Skill ツールの /xxx 起動禁止。再入系はさらに「コード実装・PR 操作・/xxx・`! <command>`
  を含む作業項目は行単位で (検証環境につき省略)。正本が地の文で指示する取得・diff・記帳は実施」
- 検証 doc: 自分がアクセスできる任意の PRD 形式 gdocs を使い、doc ID とタブ ID
  (`t.<tabID>`) を差し替える。doc により実測値は変わる

## 失敗パターン台帳 (I〜L 検証、2026-07-12)

- **前提付きフォーマットの縮退未定義**: 状態機械・記帳書式が持つ前提 (リポジトリ・戻り先フェーズ) が
  欠けるケースの読み替えが未規定で裁量補完が発生 (iter 1: スライス状態 5 値の縮退 / 再入ログの
  `→フェーズ<N>` 読み替え)。縮退規則の明文化で解消
- **ループ内終了処理の単位未定義**: フェーズ3 の Step Z 適用単位 (1 周 vs 全体) が読めない (iter 1)。
  「1 スライス = 適用単位」の明記で解消。ただし定義を置いた後、依存する下位手順の条件文
  (「フェーズ3 完了時」) の単位を書き直し忘れて iter 5 で再検出 — 単位を読み替える定義を置いたら
  依存箇所も同じ単位で書き直す
- **行内注記と一般規則の優先関係**: 「常に実行」注記が非該当判定を上書きするように読める (iter 2)、
  複数の非該当根拠の主根拠選択 (iter 5/8/9 と 3 回再発して初めて修正)、3 記法と非該当判定の順序
  (iter 10)。「注記は条件との関係のみ・非該当判定に先立たれる」「構造的判定を主根拠」で解消
- **要約ラベル・例示句の参照先未定義**: fast path の「品質パス」が指す行範囲 (iter 6)、
  「正本は .latest」の literal/プレースホルダ (iter 2) など、ラベルと実体の対応づけが推測任せ。
  参照先の明示で解消
- **同語異物の用語衝突**: 簡易プラン導入後、「プランファイルに着手行は書かない」の対象が主プランのみか
  曖昧 (iter 8)。「案件プランファイルにも簡易プランにも」への限定で解消。iter 11 で executor が
  この規則文を根拠に誤記帳を自己訂正しており、ガードレールとして機能
- **修正不要と判定した指摘** (いずれも挙動を変えないラベリング・記録粒度の揺れ):
  非該当ラベル文言と根本理由の乖離 / spike 等の抽象動詞の一般判定基準 (分割例が既に固定) /
  非該当句に依存する後続文書行の独立判定 (全 executor が一貫して独立適用) /
  fast path 各 skill のリポジトリ前提性は各 SKILL.md まで遡って確認する運用 (executor 側の実践)

## 失敗パターン台帳 (E〜H 検証、2026-07-11)

- **status 質問の実行誤爆**: 「〜だっけ？」を実行依頼と解釈しフェーズ実行+偽完了記帳
  (改良前ベースライン)。status モード新設 + 手順2 の振り分けで解消
- **探索範囲の解釈ぶれ**: 「配下」を maxdepth 1 と解釈し候補を見落とし (iter 4 critical)。
  「再帰 — 深さを絞らない」明文化で解消
- **記帳先の分裂**: 「末尾に追記」の字面適用で進捗行が `## 進捗` 節と物理末尾に分裂
  (iter 2)。「進捗欄 = `## 進捗` 節末尾」の用語定義で解消
- **条件付き詳細度の括弧注記**: 「(最終行は日付と要点も)」型の注記が両読みを生む
  (iter 4→5 で言い換えても再発)。条件性の削除 (「各行とも転記」) で解消 — 言い換えでなく
  自由度をなくすのが正解
- **軽量比較の基準欠如**: 既定候補の「最新」判定で候補の中身を深読み (iter 5 で 151K
  tokens)。「進捗欄の最終日付、無ければ mtime。中身は深読みしない」で解消 (iter 6 で 46% 減)
- **修正不要と判定した指摘**: 定義の前方参照 (DRY と衝突) / 「軽量」のさらなる線引き
  (過剰仕様化) / 規約導入前データへの适用は但し書きで対応済み
