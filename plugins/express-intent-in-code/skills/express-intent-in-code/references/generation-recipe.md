# 生成時レシピ (経路2) — コードを書いている最中の3つの瞬間 (SSOT)

適用対象: **これから書く・いま書いているコードのみ**。既存 working code の改名・変換は経路1 ([decision-procedure.md](decision-procedure.md) Step 0〜9) へ。生成時経路では既存行の改名を行わない (Surgical Changes の継承)。

コメントを減らすこと自体は目的ではない。**意図の置き場所を正しく配る** — コードで表現できる意図は名前・型・構造へ、コードから読めない why は「それを担う名前付き定義の直上」へ1箇所 — のが目的である。

## 瞬間1: コメントを書きたくなった

書く前に「どこに・何として置くか」を次の順で決める。

1. 残置4類型 — (a) 外部仕様・他システム前提 / (b) トレードオフ・最適化見送りの実測根拠 / (c) 危険・順序依存・実行時不変条件・セキュリティ上の非自明な判断 / (d) FIXME (理想+妥協理由) — に該当する → **書く。削らない。** 置き場所は、その判断を担う**名前付き定義** (メソッド / クラス / 型 / 定数) の直上に1箇所。判断が公開メソッド本体に直接書いた式に宿っているなら、**先にその式を述語 / ヘルパーへ抽出してから**、抽出先の定義直上に置く。呼び出し側 (公開本体) には同じ why を書かない — 述語 / ヘルパーの名前に語らせる。
   - **複数の隣接定義が1つの判断を共同で担う場合** (例: `selectedFieldIdRef` の存在理由 + `selectField` がその ref を書き換える手順、両方が「はね返り抑止」という1つの why を分担する): why は**動機が最も読めない側 = 本体を持つ関数の実装直上**に固定する。データ側 (ref・型・定数) の定義は名前に語らせ、コメントを重複させない。両方に書くと同一 why の重複判定 (セルフチェック4) に触れる。
2. 該当しない → コメントでなくコードに移してから先へ進む ([comment-keep-vs-promote.md](comment-keep-vs-promote.md) の昇格先表と同一):
   - 値の正体の説明 → 値オブジェクト / 型名
   - 用途・存在理由の説明 → メソッド名 (目的名)
   - 分岐理由の説明 → 述語メソッド / 説明変数
   - マジック値の意味 → 名前付き定数
   - 手順の説明 → private メソッド (機構名)

   コードを見た人なら誰でも書ける内容 (名前・型・シグネチャの言い換え) は、どこにも書かない。
3. 移すのを試みて名前・型で表現しきれなかった → コメントを書いてよい。

## 瞬間2: 手段 (機構) や制約対応を書き始めようとしている

公開メソッド / 公開関数の本体は**目的名の呼び出し列とガード節**だけになるのが完成形。異常系のガード節 (早期 return) は公開本体に置いてよく、ガード条件が複合になったら述語名を付ける。機構 (DOM 操作・座標計算・低レベル API・文字列処理) と外部制約 (フレームワーク・コマンド・既存 API の事情) への対応は、目的名の private メソッド / ヘルパーに包み (T2 / T12)、弁明はその定義直上に1箇所だけ書く。

```ruby
# before: 公開メソッドに制約の弁明と機構が露出
def import
  # 外部 API はページングが無く全件を一括で返すため、1000 件ずつ分割して保存する
  rows.each_slice(1000) { |batch| Row.insert_all(batch) }
end

# after: 制約対応を目的名に包み、弁明は定義側1箇所
def import
  save_in_api_safe_batches(rows)
end

private

# 外部 API はページングが無く全件を一括で返すため、1000 件ずつ分割して保存する
def save_in_api_safe_batches(rows)
  rows.each_slice(1000) { |batch| Row.insert_all(batch) }
end
```

公開本体の行にコメントを添えたくなったら、それはその行を述語 / ヘルパーに抽出する合図。抽出後は名前が意図を語るので、公開本体のコメントは不要になる。

## 瞬間3: 生のデータ形状に意味を説明しそうになった

- boolean 引数の意味をコメントで補いたくなった → enum / シンボル引数にする
- null と undefined (nil と偽) の意味差を説明したくなった → 判別可能な union / 値オブジェクトにする (T6)
- 同じ Hash / tuple の形が2箇所目に流れた → 型を切る (T5)

命名の接地は経路1 と同じ: 実在の語 (課題・既存コード・UI の語) に接地し、証拠なくそれっぽい語を造らない (段4 造語ゲートは [domain-abstraction.md](domain-abstraction.md))。

## 書き終えたときのセルフチェック

1. **コードから読めない why (外部仕様・実測根拠・危険・セキュリティ判断) が、名前付き定義の直上に記録されている — 0箇所は削りすぎで不合格**
2. 公開メソッド / 公開関数の本体にコメントが残っていない (残すべき why はすべて、それを担う名前付き定義の直上へ)
3. 名前・型・シグネチャの言い換えコメント: 0件
4. 同一 why の重複: 0件 (2箇所目以降は名前に語らせるか参照)
5. boolean / 2値の意味差をコメントで説明している箇所: 0件
6. 公開本体に裸の複合条件ガードが残っていない (複合になったら述語名を付ける)

## 利用側コメント規約との関係

このレシピは「コメントを書くか・どこに置くか」を決める。書くと決めた why の文面の質 (ゴール志向・根拠数値・ADR 集約・読者は plan を持たない前提) は、利用側プロジェクトのコメント規約 (例: コードコメント7原則) と [comment-keep-vs-promote.md](comment-keep-vs-promote.md) の純化ルールに従う。両者は競合しない: レシピが「所在」、規約が「文面」を担う。
