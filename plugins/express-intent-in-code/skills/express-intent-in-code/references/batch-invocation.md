# バッチ / パイプライン起動時の扱い (単一対象が未指定のとき)

`/simplify` 等の品質パイプライン後段で、単一対象も `/review-code-quality` からの naming/凝集 handoff も無いまま本スキルが起動された場合の手続き。SKILL.md「When to use / not」からの 1 hop 詳細。

## no-op 宣言か handoff 消化か

同一実装セッションで経路2 を適用済みでも、品質パス連鎖内での本呼び出しは経路1 (申し送りファイル `quality-review-handoff-<branch>.md` の消化) として独立に実行する — 経路2 実施済みを理由に no-op としない。`/simplify` 等の品質パイプライン後段で、単一対象も `/review-code-quality` からの naming/凝集 handoff も無いまま起動された場合 — diff 全体を改名候補スキャンしない (それは `/review-code-quality` の役割で、本スキルが肩代わりしない)。handoff があればそれを変換対象に取り、handoff も明示対象も無ければ diff 名を列挙せず「handoff 無しのため変換対象なし」と即 no-op 宣言する (無人パイプラインを確認待ちで止めない)。

## 例外 — 機械検査で拾える 3 兆候は handoff の有無によらず対象に加える

(a) 多義衝突 (blast radius 内で同じ語が 2 つのドメイン概念を指す。例: URL 認証の `token` と焼き込みプレースホルダーの `token`)、(b) 新規識別子の段0 ノイズ語 (`doc`/`data`/`info`/`target`/`tmp` 等)、(c) 新規/移植ファイルのコメント過剰 — 非空行に占めるコメント行の割合が目安 25% を超えるファイルを Step 6 (コメントの蒸留と純化) の対象に加える (閾値は超過検知の目安。残置4類型で正当に密なファイルは維持してよい。実測根拠: PoC 移植で移植元のコメント密度 — 機構リプレイ・セクション目次を含む — がそのまま残り、handoff 無しでは素通りして利用者の手動指摘で発覚した)。

診断器の handoff は表層 finding に偏りこれらを落とす実績があるため、機械的な 1 パス ((a)(b) は grep、(c) は行数カウント) に限り本スキルが直接スクリーニングする (診断的な網羅レビューには広げない)。スクリーニングや変換中に見つけた**副次候補** (対象外の別衝突・別ノイズ語) は対象化せず、出力の据え置きログに 1 行で残して次の `/review-code-quality` パスの入力にする。

## ファイル改名 (mv) を伴う変換

**ファイル改名 (mv) を伴う変換は適用直前に handoff / ledger を再読する** — 並行アクタ (別セッション・再開された subagent) が据え置きや別決定を記帳していたら、その決定に従い変換を中止する (理由: 据え置きの記帳と並行して mv を進め、改名が中途半端に残って import 破壊と全量巻き戻しに至った実測。Edit の変更検知と違い mv は衝突を検出しない)。
