# コメント keep-vs-promote 決定表 (SSOT)

why コメントを「名前/型/構造/テストへ昇格して削除する」か「コードから絶対に読めない真の why として残す」かの判定。**ゴールはコメント撲滅でなく純化** — 昇格できる why は移して消し、残すべき why は文面を良くして残す。

判定軸はただ一点: **その情報はコードの抽象 (名前・型・構造・テスト) から自明か** (Ousterhout「Comments should describe things that aren't obvious from the code」)。自明にできるなら昇格、できないなら残す。

> このファイルは配布される skill の自己完結 SSOT。一般に流布する「コメントは Why を書く / 自明な What は書かない / 同一 Why を散らさず ADR 集約 / 読者は plan を持たない前提」という原則と整合させてある。

## 昇格して削除する (コードへ移せる what / 名前化できる why)

| コメントの種類 | 昇格先 | 例 |
|---|---|---|
| 値の正体の説明 | 値オブジェクト/型名 | `# 検証済み座標` → `SignatureAnchor` 型 |
| 存在理由・用途の説明 | メソッド名 (目的名) | `# 署名配置に使う` → `signature_anchor_boxes` |
| 分岐理由の説明 | 述語メソッド名/説明変数 | `# 本文領域内か` → `in_body_region?` |
| マジック値の意味 | 名前付き定数 | `# 0.8 = 足切り` → `MIN_RELIABLE_CONFIDENCE` |
| 計算/変換手順の説明 | private メソッド (機構名) | `# PDF 座標に変換` → `to_pdf_coords` |
| 自明な what | (削除) | `# ユーザー名を返す` (`get_user_name` の上) |
| シグネチャの言い換え | (削除) | 名前・型・戻り値と一致する冗長コメント |

**順序厳守**: 先に名前/構造を変換し、表明できた why を**後から**削除する。コメントを先に消すと、名前がまだ用途を表明できていない段階で背景を失う。

## 残す = コードから絶対に読めない真の why (4 類型)

以下に該当したら**昇格を試みず残す**。文面は「ゴール志向化 (何を達成したくてこの実装か) + 根拠数値付与」だけ改善する。

| 類型 | 内容 | 例 |
|---|---|---|
| **(a) 外部仕様・他システム前提** | 自コードからは読めない外部の事実 | `# PDF は左下原点。文字座標が無いので xhtml が唯一の出所` / `# この API はソート保証なし` |
| **(b) トレードオフ/最適化見送りの根拠数値** | なぜこの手段を選び、いつ見直すか | `# 月次200件のため同期で許容。1万件超で Sidekiq 移行` / `# 実測 320/480 語、上限 500 で確保` |
| **(c) 危険・順序依存・実行時不変条件** | どんな関数名にも込められない制約 | `# この 2 行を入れ替えると署名者が欠落する` / `# 呼ぶ前に lock 必須` |
| **(d) 将来予定 FIXME** | 理想の実装 + 妥協理由 | `# FIXME: v2 API リリース後にリトライ削除` |

## 純化ルール

- **構造的な Goal→制約→手段は ADR に集約**し、コメントは「ゴール要約 1 文 + 参照」に縮約 (`# 設計詳細: docs/adr/0004-....md`)。同一 why を 3+ 箇所に長文で散らさない。
- **テスト/property に why を移したら**、production の冗長コメントは「仕様はテスト参照」1 行に縮約し SSOT が割れるのを防ぐ。
- **残す why の文面**は「plan / 設計メモを読まない reviewer / future-self」が codebase だけで読み下せる語で書く。plan 内造語・番号ラベル・section anchor は持ち込まない (grep / 公開 tracker で辿れる identifier のみ可)。日本語の地の文に英語の慣用句・業界スラングを混ぜない。

## polish-before-commit との棲み分け

`/polish-before-commit` のコメント整形は**プロジェクト規約テキスト依存の機械適用** (規約ファイルに書かれた形式へ揃える)。本節は**対象 1 点の昇格判断** (この why はこの名前/型に移せるか) に特化する。前者は形式、後者は意図の所在。
