# 命名梯子 (Naming Ladder)

名前を 1 段ずつ目的 (why) に近づける手続き。Arlo Belshee「Naming as a Process」の段階ラダーを、機構名→what名→目的名の昇格に再構成したもの。**飛び級は禁止** — 段0 からいきなり目的名を考えると、機構違いの別案が複数出て発散し、caller の実際の用途と無関係な「それっぽい名前」をでっち上げてしまう。

## 各段の定義・見分け方・上げ方

### 段0 Missing / Nonsense (機構名)

- **特徴**: 名前の核が「取得元」「内部表現」「ノイズ語」でできている。実装の how が名前に漏れている。
- **機構語リスト**: 取得元 (`from_xhtml` `xhtml` `from_pdf`)、内部表現 (`bbox` `json` `hash` `dto` `vo` `impl`)、ノイズ語 (`data` `info` `result` `value` `obj` `item`)、汎用動作語 (`manager` `util` `handle` `process` `do`)、捨て値 (`tmp` `temp` `retval` `ret`)。
- **上げ方**: 名前と実態の偽情報をまず正す。how 由来語 (取得元・内部表現・ノイズ語) を棚卸しして除去候補に印を付ける。ただし**消すのは public 名からだけ** — 機構語は private メソッド/構築経路に残し、パースのバグを追う人が grep で辿れるトレーサビリティを保つ。

### 段1 Honest (what 名)

- **特徴**: やっていること・戻り値の形は読める。だが「何のために必要か (目的)」は読めない。`word_coordinate_data` は「単語の座標データ」= 形状は分かるが用途が不明。
- **上げ方**: 目的名へ**直行しない**。いったん「何を・何の入力から・何を経由して得るか」を接続詞込みで全部出した**長い正直名**へ改名する。例: `each_word_box_in_page_coords_for_overlaying_signature_fields`。末尾に滲み出る `for_<目的>` を捕まえる。醜くて長くてよい — それが次段の構造発見の材料になる。

### 段1.5 構造の発見 (長さを読む)

- **特徴**: 長い正直名は「名前の問題」ではなく「**構造欠如のサイン**」。
  - `and` / `with` が複数ある = **責務が複数** → メソッド分割 (T2)。
  - `coordinate` / `data` のような機構語が**どうしても消せない** = **ドメイン型が欠落** → 値オブジェクト/クラス抽出 (T5)、または状態の sum type 化 (T6)。
- **上げ方**: 語を削って短くするのではなく、**構造を直して副産物として名前が縮む**ようにする。`word_coordinate_data` の `coordinate`/`data` が消せないのは、座標の集まりに名前 (型) が無いから。`SignaturePlacementMap` のような型を導入すれば、メソッドは `signature_anchors` まで縮む。

### 段2 Does-the-Right-Thing (嘘の除去)

- **特徴**: 名前に出ていない**副作用・前提・除外**がある (空行除外・座標系反転・キャッシュ・ソート)。名前が嘘をついている限り、読み手は why コメントでしか真実を知れず、これがコメント依存の根本原因。
- **上げ方**: 隠れた挙動を列挙し、(a) 名前へ昇格 (`signature_anchors_excluding_empty_lines`) するか、(b) 別メソッドへ追い出して名前 = 契約を成立させる。「名前が約束する範囲」と「実際にやること」を一致させてから次段へ。

### 段3 Intent (目的名)

- **特徴**: 呼び手にとっての**ドメイン上の役割 (why)** を表明する。`signature_anchor_boxes` = 「署名フィールドを配置する錨点」= 何のために存在するかが名前で分かる。
- **上げ方**: 全 caller を grep し「戻り値を次に何に使うか」を 1 動詞句で言語化 → その目的を名前の核に据える。複数 caller が**別目的**なら、それは 1 メソッド 2 役のサイン。改名でなく**分割**が正解 (1 つの名前に 2 つの目的は載らない)。

### 段4 Domain-abstraction (ユビキタス言語) — 既定の到達目標

- **特徴**: ドメイン専門家が会話で使う**実在の共有語**へ。`signing_positions` のように、PR・設計書・専門家との会話・UI 文言に出てくる語。
- **上げ方は 2 経路**: (経路A) grep/仕様/UI 文言に実在語があれば**それへ snap** (自分で synonym を作らない)。(経路B) 実在語が無いが primitive が群れている (同一パラメータ集合・名前が型/値を狭める `bbox: Hash`) なら、**欠落した型 (Whole Value) を抽出**して周辺の世話コードを移す (T5/T6)。段4 は辞書引きの改名ではなく、しばしば**型抽出という構造変更**。
- **目標の扱い**: 段4 は**可能な限り到達する既定目標**。対象ごとにドメイン語の探索手続き (Step A–H) を必ず実行し、genuine-vs-invented ゲートが通れば段4 へ昇格する。段3 据え置きは**探索が空振りした証拠 (探索ログ) を残したときに限る** (根拠ある fallback)。「段3 で足りるから」を据え置きの既定にしない。
- **造語の歯止め**: 共有語が無いのに「それっぽい」語を作って段4 に上げない (= `/purge-private-vocab` が消す造語を逆に生む)。plausible は証拠でない。判定に迷えば下位段に留める (誤誘導名は無意味名より有害)。
- 到達手続き・探索 Step A–H・ゲート・据え置き記録の全文は [domain-abstraction.md](domain-abstraction.md)。

## bbox_xhtml ケースの全段ウォークスルー

ユーザーの実例 (ninja-sign, Ruby) を梯子で歩く。

| 段 | 名前 | なぜこの段か | 次段へ何をしたか |
|---|---|---|---|
| 段0 | `bbox_xhtml` | `bbox` (内部表現) + `xhtml` (取得元) = 機構語のみ。何用か皆無 | `xhtml` を private 構築経路へ退避、偽情報除去 |
| 段1 | `word_coordinate_data` | 「単語の座標データ」形状は読めるが目的不明 | 長い正直名 `each_word_box_in_page_coords_for_placing_signature_fields` へ展開 |
| 段1.5 | (長い正直名) | `for_placing_signature_fields` が目的、`coordinate_data` が消せない = 座標集合の型欠落 | 座標集合を値オブジェクト化 (T5) する判断 |
| 段2 | (嘘の発見) | 本体で空行を除外しているが名前に出ていない | `_excluding_empty_lines` を名前 or 別メソッドへ |
| 段3 | `signature_anchor_boxes` | caller が戻り値を「署名フィールドの配置 (押印アンカー)」に使う = ドメイン役割 | caller 用途を grep で確認して確定 |
| 段4 | `signing_positions` | 探索手続きで「署名位置/signing position」を仕様・UI 文言・sibling 型に発見 (≥2 ソースで grounding) → 経路A で snap。発見できなければ探索ログを残して段3 据え置き | Step B–D を実行し file:line を引用 |

**残す真の why** (どの段でも昇格できない): 「PDF は左下原点・文字座標が無く xhtml が唯一の出所」は外部仕様 (4 類型 a) なので、どんな名前にも込められない → コメントとして残す (純化のみ)。

> 段4 を既定目標として**毎回探索する**。段0/段1 で止めない (機構名/what 名で満足しない) のは当然として、段3 で止める時も「探索したが実在語が無かった」という探索ログを残す。探索せずに段3 で安住するのが、本体を読まないと目的が掴めない名前を残す退行の正体。
