# AC enumerate ルール (Step 0-3 詳細)

`## 受け入れ条件` セクションを行単位でパースし、`- [ ]` 行に AC-ID (`AC-1, AC-2, ...`) を機械的に付与する。

## 正規化形式 (全カテゴリ統一)

元の `- [ ] permission: ...` 行を **全カテゴリで観点ラベルを内部メタデータに展開** する。subagent 側のパース分岐を避けるため非対称扱いは禁止。

- 正常系 / 異常系:
  `- AC-N (カテゴリ, 観点: <ラベル>): 本文`
  例: `- AC-1 (正常系, 観点: permission): 管理者が PATCH ... → 200 OK`
- エッジケース:
  `- AC-N (エッジケース, 観点: <ラベル>, 境界値: <値>): 本文`
  例: `- AC-5 (エッジケース, 観点: permission, 境界値: 本人): 管理者が自分の role を PATCH → 422`
- 非影響確認: 正常系と同形式

### ルール

- チェックボックス `[ ]` は除去
- AC-ID は連番
- 観点ラベル `permission:` プレフィックスは展開済みなので本文側からは削除
- 「正常系/異常系では観点ラベルを残す、エッジケースだけ展開」のような**非対称扱いは禁止**

## 出力例 (`${ENUMERATED_AC}`)

```
- AC-1 (正常系, 観点: permission): 既存ユーザが /auth/login → 200 + JWT
- AC-2 (正常系, 観点: permission): 新規ユーザが /auth/saml/login → IdP リダイレクト → callback → ユーザ作成 + JWT + /dashboard
- AC-3 (異常系, 観点: permission): SAML 有効ユーザが /auth/login → 403
- AC-4 (エッジケース, 観点: permission, 境界値: 本人): 本人が PATCH /api/users/self → 422
```

subagent (BB / WB / Red Team) はこの正規化済み文字列を AC-ID ソースとして使い、元の `- [ ]` 行を再パースしない。

## 上流 skill (`/define-acceptance-criteria`) 出力フォーマット契約

| 期待フォーマット | 違反時の挙動 |
|---|---|
| 分析ファイルに `## 受け入れ条件` (見出しレベル `##`) | セクション無し → Step 0-2 のエラーメッセージで中断 |
| カテゴリは `### 正常系` / `### 異常系` / `### エッジケース` / `### 非影響確認` の `###` 見出し | カテゴリ見出し無し → 全項目を `カテゴリ:不明` で進める |
| AC 行は `- [ ]` 行頭 (チェックボックス必須) | `- ` のみ (チェックボックスなし) → 同様に enumerate (`- [ ]` `- ` 両方を許容) |
| エッジケース行は `[観点ラベル] [境界値: カテゴリ名]:` 形式 | 形式違反 → `観点:不明` でカテゴリ保持し進める |
