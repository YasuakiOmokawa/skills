# プラン修正・追記フォーマット

MECE 分析完了後の出力先は **分析ファイル**（`{プランファイル名}.analysis.md`）。
プランファイル本文には品質検証サマリー 1 行のみ追記する。

## 修正ルール

| 重要度 | 対応 |
|--------|------|
| 🔴 Critical | 下記セクションに記録 + プラン内の該当箇所（セクション名・内容）と推奨修正内容を併記 |
| 🟡 Important | 下記セクションに追記のみ（プラン本文は変更しない） |
| 🟢 Nice-to-have | 下記セクションに追記のみ（プラン本文は変更しない） |

## 分析ファイルへの追記セクション

```markdown
## MECE分析結果

### 分析サマリー
- 分析日時: YYYY-MM-DD
- 対象リポジトリ: ${REPO_LIST}
- ACカバレッジ: N/M項目充足
- 漏れ件数: X (お見合い検出された件数 = 両 Analyst が言及ゼロで Red Team が独自検出)
- 重複件数: Y (BB ↔ WB が同じ問題を別角度で言及した「真の合意」件数)
- 判定: MECE OK / 要修正（Critical N件）

### 4分類クロスリファレンス (area タグで機械集約)
| # | Area | BB 指摘 ID | WB 指摘 ID | 分類 (真の合意/補強合意/実装漏れ/仕様漏れ/お見合い) | 統合 Severity | 統合内容 |
|---|------|------|------|------|---------|---------|

**分類の意味**:
- **真の合意**: BB と WB が同じ問題を**同じ層**から発見 (両方仕様 or 両方コード)
- **補強し合う合意**: BB と WB が同じ問題を**異なる層**から発見 (信頼度最高)
- **実装漏れ**: 仕様にあるがコードにない (BB ✓ WB —)
- **仕様漏れ**: コードにあるが仕様未記載、または暗黙の前提 (BB — WB ✓)
- **お見合い**: 両 Analyst が言及ゼロ、Red Team が独自検出

**集約は area タグで機械化**: 同じ `Area` 値を持つ BB/WB 指摘を peer 候補として並べ、Issue 本文で意味的一致を判定する。

### ACカバレッジ検証結果 (main agent が BB / WB の AC 判定から機械合成)

| AC-ID | AC項目 (短く要約) | カテゴリ | BB 判定 | WB 判定 | 総合判定 |
|---|--------|---------|--------|---------|---------|
| AC-1 | ... | 正常系/異常系/エッジ/非影響 | 充足/不十分/言及なし | 充足/不十分/言及なし | 充足/不十分 |

**機械合成手順** (main agent):
1. dispatch 時に AC を `AC-1, AC-2, ...` で序数付けて subagent に渡す
2. BB / WB が返す「AC 判定」テーブル (AC-ID + 判定) を AC-ID で join
3. red-team-checklist.md の「AC カバレッジ機械合成」ルールに従って総合判定列を算出
4. 元 AC 文の「カテゴリ」(正常系/異常系/エッジ/非影響) を分析ファイルの AC セクションから引いて補完

**ACカバレッジ N/M の定義**: M = **元 AC 項目数（`[MECE追加]` を除く）**、N = M のうち総合判定が「充足」となった項目数。`[MECE追加]` 項目は分母に含めない（既存 AC のカバレッジを測る指標のため）。

#### MECE分析によるAC追加提案
- [ ] `[MECE追加]` [追加すべきAC項目の説明]
- [ ] `[MECE追加]` ...

### Critical指摘（要修正）
| # | 指摘 | プラン該当箇所 | 推奨修正内容 | 根拠 (BB/WB/Red Team) | 4分類 |
|---|------|--------------|------------|------|---------|
| 1 | ... | セクション名・内容 | ... | BB-C1 + WB-C3 | 真の合意 |

### Important / Nice-to-have
| # | 重要度 | 指摘 | 推奨対応 | 根拠 | 4分類 |
|---|--------|------|---------|------|---------|
| 1 | 🟡/🟢 | ... | ... | ... | ... |

### お見合い検出 (Red Team 独自)
| # | 領域 | チェックリスト観点 | 発見事項 | Severity |
|---|------|---------|---------|---------|

### 純技術リスク補完 (Red Team が BB/WB 盲点を補完)
| # | 観点 (セキュリティ/パフォーマンス/依存) | 発見事項 | Severity |
|---|------|---------|---------|

### Red Teamレビューサマリー
<details>
<summary>Red Team 統合評価レポート全文</summary>

${RED_TEAM_RESULT}

</details>

### 各ロール分析詳細
<details>
<summary>BB Analyst 分析結果</summary>
${BB_RESULT}
</details>

<details>
<summary>WB Analyst 分析結果</summary>
${WB_RESULT}
</details>

<details>
<summary>Wiki Researcher 参考情報</summary>
${WIKI_RESULT}
</details>
```

## プランファイルへのサマリー追記

プランファイルの `## 品質検証` セクションに **1 行だけ** 追記する（セクションがなければ作成）:

```markdown
- MECE判定: [OK or 要修正（Critical N件）] / ACカバレッジ [N]/[M]項目 / 漏れ [X]件 / 重複 [Y]件 → [分析ファイル名]
```

## AC ブラッシュアップの運用ルール

### `[MECE追加]` タグの適用範囲

| ケース | タグ | 書き方 |
|---|---|---|
| 新規 AC 項目を追加 (仕様漏れ・お見合いから) | `[MECE追加]` | 新しい行頭に `[MECE追加]` を付け、対応するカテゴリ（正常系 / 異常系 / エッジ / 非影響）の **既存セクション内** に追記 |
| 既存 AC を具体化（補足を加えるだけ） | タグ不要 | 元 AC 行の末尾にカッコ書きで補足 |
| 既存 AC を修正（書き換える） | `[MECE追加 変更]` | 修正後の行頭に `[MECE追加 変更]` タグを付け、修正理由を併記 |

**専用サブセクション（`### [MECE追加] 新規AC項目` 等）は作らない**。カテゴリ分類が崩れるため、必ず既存カテゴリに追記する。カテゴリに該当しない横断的な項目は分析ファイル末尾に `### その他（[MECE追加]）` を新設して追記。

### 「充足」判定の基準 (機械的に決定)

ACカバレッジ表の「総合判定」列は、BB / WB の判定から機械的に決定する (メインエージェントが計算):

```
BB 検証列と WB 検証列を見て:
├─ どちらかに「不十分 / ❌」あり → 総合判定: 不十分
├─ 両方が「言及なし」 → 総合判定: 不十分 (Red Team お見合い対象)
└─ 少なくとも一方が「充足 / ✅」、かつ もう一方は「充足」または「言及なし」
    → 総合判定: 充足
```

**両ロール「充足」要件にしない理由**: BB は仕様観点、WB は実装観点で見ているため、片方が言及していなくても、もう片方が充足判定なら「不十分」とは言えない。Red Team が独自に懸念検出した場合のみ「不十分」に格下げ。

### 「カバレッジ不十分」と「AC 自体の不備」の分類フロー (4 分類との対応)

```
発見した問題は何か?
├─ AC は明確だが、プラン / 実装方針で対応されていない (BB ✓ WB — = 実装漏れ)
│   → 「Critical 指摘」に記録 + プラン側を修正
├─ AC の記述が曖昧、または重要要件が暗黙 (BB — WB ✓ = 仕様漏れ)
│   → 「AC ブラッシュアップ」で `[MECE追加]` または既存 AC 具体化
├─ 両者言及ゼロで Red Team が検出 (お見合い)
│   → 「AC ブラッシュアップ」で `[MECE追加]` + Critical 候補なら指摘も記録
└─ BB と WB が同じ問題を別角度で発見 (真の合意)
    → 「Critical 指摘」に統合 1 件として記録 (重複カウントしない)
```

判別の例:
- 「過去版をダウンロードできる」AC があり、プランに権限チェック実装の記述なし (BB ✓) → コードに権限チェックなし (WB ✓) → **真の合意 Critical**
- 「DB トランザクションでロールバック」AC があり、コードに transaction なし (BB ✓ WB —) → **実装漏れ Critical**
- コードに soft-delete scope があるが AC に削除済ユーザの扱い記述なし (BB — WB ✓) → **仕様漏れ、AC に `[MECE追加]`**
- BB / WB 両者言及ゼロの XXE 攻撃を Red Team が検出 (お見合い) → **AC に `[MECE追加]` + Critical 指摘**
