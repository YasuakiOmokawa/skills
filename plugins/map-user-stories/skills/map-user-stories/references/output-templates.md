# 出力テンプレート

**セクションヘッダー・カラム名・コードフェンス形式を変更しない**。`create-jira-issues` がこの形式でパースする。

## 出力セクション構成（この順序で出力）

```
## Context
## Phase 0: {Phase名}
## Phase 1: {Phase名}
## Phase N: {Phase名}
## スプリントマッピング
## Jira ↔ US マッピング
## タスクリスト
## US TSV
## 未解決事項
```

### `## Context`

```markdown
## Context

- プロジェクト名: {名前}
- 目的: {1-2文}
- チーム人数: {N名}
- スプリント長: {N週間}
- 開発期間: {N週間}
- 入力ソース: {設計書/Jira/Confluence等}
```

### `## Phase N: {Phase名}`

Phase分類の基準:
- **Phase 0**: 前提条件・インフラ（他Phase全てが依存する基盤）
- **Phase 1**: コア機能（MVP。これがないとプロダクト価値がない）
- **Phase 2**: 移行・既存対応（既存データ/フロー/ユーザーへの対応）
- **Phase 3**: 外部連携・収束（外部サービス連携、仕上げ）
- **後続**: 別DD/別計画で扱うスコープ外

**注意**: Phase番号と名称はプロジェクト特性に応じてカスタマイズ可。上記は例。

各Phaseに以下のMarkdownテーブルを記載:

```markdown
## Phase 1: コア機能

| US_ID | ユーザー | ストーリー | 受入条件 | 依存US | Jira | 技術メモ |
|-------|---------|-----------|---------|--------|------|---------|
| US-001 | {ペルソナ} | {〜したい。〜のため。} | {AC1 / AC2 / ...} | US-XXX | | {実装上の留意点} ／ やらない: {スコープ外項目} ／ 対象外: {別 US 範疇} |
| US-002 | ... | ... | ... | | | |
```

**技術メモ内のプレフィックス記法**: `やらない: ` / `対象外: ` で始まる行は、`create-jira-issues` により Jira description の「やらないこと」セクションへ転記される。1 セル内で複数行ある場合は ` ／ ` で区切る（Markdown テーブルの行内改行は不可のため）。

#### カラム仕様

| カラム | 必須 | 形式 | 説明 |
|--------|------|------|------|
| US_ID | ✅ | `US-{3桁連番}` (例: US-001) | 全Phaseで通し番号 |
| ユーザー | ✅ | ペルソナ名 or ロール名 | 誰の視点のストーリーか |
| ストーリー | ✅ | 「〜として、〜したい。〜のため。」形式 | ユーザー視点で記述 |
| 受入条件 | ✅ | `AC1: ... / AC2: ...` スラッシュ区切り | ユーザー視点の完了条件。技術的TODOにしない |
| 依存US | | `US-XXX` カンマ区切り | このUSの前に完了すべきUS |
| Jira | | チケットキー or 空文字 | 既存チケットがあれば記入。`create-jira-issues` が書き戻す列 |
| 技術メモ | | 自由テキスト + プレフィックス記法 | 実装上の留意点、既存コードパターンへの参照等。**具体ファイルパスは設計が進むと stale 化するため最小限**。コンポーネント名・ディレクトリレベルに留める。例外: プロトタイプから得た decision-encoding snippet（state machine / schema / type shape）はインライン化可。**`やらない: ` / `対象外: ` プレフィックス行で「やらないこと」を記述**（`create-jira-issues` が抽出して Jira description の「やらないこと」セクションに転記する） |

### `## スプリントマッピング`

```markdown
## スプリントマッピング

| Sprint | 期間 | US | タスク | ゴール |
|--------|------|----|--------|--------|
| S1 | Week 1 | US-001, US-002 | T-001〜T-005 | {このスプリントの達成目標} |
| S2 | Week 2 | US-003 | T-006〜T-010 | ... |
```

### `## Jira ↔ US マッピング`

```markdown
## Jira ↔ US マッピング

| 既存Jira | 対応US/タスク | 状態 |
|----------|-------------|------|
| XPROJ-100 | US-001, T-001 | Open |
| XPROJ-101 | US-002 | In Progress |
```

### `## タスクリスト`

TSVコードフェンスで出力（`create-jira-issues` パース対象）:

````markdown
## タスクリスト

```tsv
US_ID	Task_ID	タスク名	説明	依存タスク	Jira	備考
US-001	T-001	{タスク名}	{説明} ／ 完了条件: {独立完了条件があれば}		 	やらない: {スコープ外} ／ 対象外: {別タスク範疇}
US-001	T-002	{タスク名}	{説明}	T-001	 	{備考}
US-002	T-003	{タスク名}	{説明}	T-001,T-002	 	{備考}
US-003,US-005	T-004	{複数 US を 1 タスクに統合する例: 同一 view template の編集で複数 US の AC を独立検証できる場合}	{説明}	T-002	 	やらない: {スコープ外}
```
````

**説明・備考内のプレフィックス記法**:
- **説明**: `完了条件: ` / `AC: ` 行で独立した完了条件を記述。無い場合は `create-jira-issues` が親 US の AC を継承する
- **備考**: `やらない: ` / `対象外: ` 行で「やらないこと」を記述
- 1 セル内で複数行ある場合は ` ／ ` で区切る（TSV の行内改行は不可のため）

#### カラム仕様

| カラム | 必須 | 形式 | 説明 |
|--------|------|------|------|
| US_ID | ✅ | `US-{3桁連番}` | 親ストーリーのID |
| Task_ID | ✅ | `T-{3桁連番}` (例: T-001) | 全USで通し番号 |
| タスク名 | ✅ | 短い名前 | Jira summary に使われる |
| 説明 | ✅ | 自由テキスト + プレフィックス記法 | Jira description の元。**`完了条件: ` / `AC: ` プレフィックス行で独立した完了条件を記述可**（無い場合は親 US の AC を継承） |
| 依存タスク | | `T-XXX` カンマ区切り | このタスクの前に完了すべきタスク |
| Jira | | チケットキー or 空文字 | `create-jira-issues` が書き戻す列 |
| 備考 | | 自由テキスト + プレフィックス記法 | Ready条件、補足等。**`やらない: ` / `対象外: ` プレフィックス行で「やらないこと」を記述**（`create-jira-issues` が抽出して Jira description の「やらないこと」セクションに転記する） |

### `## US TSV`

コピペ用TSV（スプレッドシート貼り付け用）:

````markdown
## US TSV

```tsv
US_ID	Phase	ユーザー	ストーリー	受入条件	依存US	Jira	技術メモ
US-001	Phase 0	管理者	〜したい	AC1 / AC2		 	留意点
US-002	Phase 1	ユーザー	〜したい	AC1	US-001	 	留意点
```
````

### `## 未解決事項`

```markdown
## 未解決事項

- [ ] {未決定事項1}
- [ ] {確認が必要な事項2}
- [ ] {ステークホルダー確認待ちの事項3}
```

## 空セクション・該当なし時の書き方

`create-jira-issues` が後段でパースするため、**下記ルールで書き方を統一する**。揺らぎを許容すると後段のパースが壊れる。

### Phase N セクション

- 該当 US が 0 件の Phase は **セクションごと省略してよい**（空テーブルを残さない）
- Phase 番号は詰めない（例: Phase 2 該当 US なしの場合、`## Phase 0 / ## Phase 1 / ## Phase 3` のように元番号を維持）
- 最低 1 つの Phase セクションは必ず出す

### `## Jira ↔ US マッピング` が空のとき

既存 Jira が無くてもセクションは出力する。テーブルは次の固定 1 行を使う:

```markdown
## Jira ↔ US マッピング

| 既存Jira | 対応US/タスク | 状態 |
|----------|-------------|------|
| （既存なし） | - | - |
```

### `## 未解決事項` が空のとき

全て解決済みでもセクションは出力する。空リスト `- (なし)` の 1 行で可。

## TSV フォーマット注意

- **純粋タブ区切り**。カラム区切りにスペースを混ぜない
- 空セルは **タブ同士を直接連続**（`\t\t`）させる。本ドキュメントのサンプル中に見える `	 	`（タブ+半角スペース+タブ）は Markdown での視認性確保のための便宜表記。実際のコピペ元は `\t\t` を使う
- 行末にタブ・余白を付けない
- 全ての行で カラム数（US TSV は 8 列 / タスクリストは 7 列）を厳密に揃える

## Markdown テーブル空セルの表記

Phase N セクションの US テーブル（Markdown 形式）で**空セルがある場合は半角スペース 1 つ**を置く。

- ✅ `| US-001 | 受領者 | ... | AC1 / AC2 | | | 技術メモ |` （依存US と Jira が空、Markdown の `| |` でレンダラーが列幅を潰さない最低保証）
- ❌ `| US-001 | 受領者 | ... | AC1 / AC2 |||技術メモ |` （`|||` は一部レンダラーで列ズレ誤認の原因）

「依存US」「Jira」列が無い US で誤って情報を省略させないため、レビュー時にも空セルとして明確に視認できるようにする。

## create-jira-issues の 4 セクション契約

`create-jira-issues` は各 US / タスクの Jira description を **4 セクション固定**（`着手条件 / やること / やらないこと / 完了条件`）で生成する。本テンプレートの各列が以下のように転記される:

### US（ストーリー）の転記マッピング

| Jira description セクション | 抽出元（プラン列） | フォールバック |
|---|---|---|
| 着手条件 | 「依存US」→ Jiraキーに展開 | 「（なし）」 |
| やること | 「ストーリー」 | （必須） |
| やらないこと | 「技術メモ」内の `やらない: ` / `対象外: ` 行 | 「（特になし）」 |
| 完了条件 | 「受入条件」 | （必須） |

### タスク（サブタスク）の転記マッピング

| Jira description セクション | 抽出元（プラン列） | フォールバック |
|---|---|---|
| 着手条件 | 「依存タスク」→ Jiraキーに展開 | 「（なし）」 |
| やること | 「タスク名」+「説明」要点 | （必須） |
| やらないこと | 「備考」内の `やらない: ` / `対象外: ` 行 | 「（特になし）」 |
| 完了条件 | 「説明」内の `完了条件: ` / `AC: ` 行 | 「親US {Jiraキー} の AC に準拠」 |

### 4 セクション充足チェック

出力直前に**全 US / 全タスクで以下を確認**:

- [ ] **着手条件**: 依存 US / 依存タスクが正しく書かれているか（無ければ空欄で OK、`create-jira-issues` が「（なし）」に変換）
- [ ] **やること**: ストーリー文 / タスク名+説明が空でないか
- [ ] **やらないこと**: スコープ外項目を技術メモ / 備考にプレフィックス記法で書いたか（無ければフォールバックの「（特になし）」で OK だが、**現実には必ず何かしらの『やらないこと』がある**ため、明示的に記入することを推奨）
- [ ] **完了条件**: 受入条件 / 完了条件が書かれているか（タスクは親 US AC 継承で OK）

⚠️ **「やらないこと」を全 US/タスクで空欄にしない**。Jira チケット化されたとき、スコープが曖昧で実装者が判断に迷う。最低限「次の PR でやる」「別タスク T-XXX で対応」「Phase N 以降」などを書く。

### プレフィックス記法の例

```markdown
| 技術メモ |
|---|
| controller_name は既存の DocumentsController を流用 ／ やらない: 印影変更（US-005 で別途対応） ／ 対象外: 添付ファイル管理画面（別画面） |
```

```tsv
US_ID	Task_ID	タスク名	説明	依存タスク	Jira	備考
US-001	T-001	文書情報コンポーネント実装	component + template + spec を新設 ／ 完了条件: 既存画面の表示が変わらない		 	やらない: スタイル変更（T-003 で対応） ／ 対象外: i18n 対応
```

## 受入条件の書き方ガイド

### ❌ 悪い例（技術TODO）
```
AC1: CrossTeamProjectモデルを作成する
AC2: Banken Loyaltyを設定する
AC3: Flipperフラグを追加する
```

### ✅ 良い例（ユーザー視点）
```
AC1: プロジェクト名・説明を入力して作成できる
AC2: 参加チームを検索して追加できる
AC3: フィーチャーフラグOFF時はメニューに表示されない
```

**判断基準**: 「ユーザーが画面上で確認できる動作か？」→ Yes なら OK。

### AC 数の上限

1 US あたり AC は **4個以下**。5個以上になる場合は US が大きすぎるサインなので分割する。

例: 「レイアウト変更」で AC が 5個以上に膨れたら、「見出し追加」「サムネイル撤去」「要素順変更」など独立した体験変化ごとに US を分割する。

## タスク粒度の判断（finalize-plan PR ガイドラインとの整合）

タスクリストは後段 `create-jira-issues` で Jira 化され、その後 `finalize-plan` で PR 分割される。タスク粒度を `finalize-plan` の PR ガイドライン（**2コミット以内 / 5ファイル以下 / 1つの論理的変更単位**）に合わせて出力すると、後段の PR 分割が自然に 1 タスク = 1 PR に収まる。

**原理: Tracer Bullet Vertical Slice** — 各タスクは特定ユーザー価値に対して全層（schema/API/UI/test）を貫く end-to-end な薄切り。層ごとの水平分割（model だけ・view だけ・test だけ）にしない。Vertical なら 1 PR で 1 ユーザー価値が demoable / verifiable。

### ❌ 粒度過剰（同一 PR に収まるべき要素を分解しすぎ）

```tsv
US-001	T-001	Documents::DocumentInfoComponent クラス作成
US-001	T-002	ERB template 作成
US-001	T-003	スタイル定義
US-001	T-004	単体テスト作成
```
→ 全て同一コンポーネント新規実装の一部。1 PR で完結する。

### ✅ PR 粒度に統合

```tsv
US-001	T-001	文書情報コンポーネント新規実装 (component + template + style + spec, ~4ファイル)
```

### ❌ 粒度不足（1 タスクが複数 PR 相当に巨大化）

```tsv
US-001	T-001	文書確認画面リニューアル全体（新規コンポーネント + リファクタリング + レイアウト変更 + Flipperフラグ, 10ファイル以上）
```

### ✅ PR 単位に分割

```tsv
US-001	T-001	添付ファイル一覧リファクタリング (h2見出し抽出, 2ファイル)
US-002	T-002	文書情報コンポーネント新規実装 (component + spec, 3ファイル)
US-003	T-003	添付ファイル一覧トグル表示対応 (controller + view + Flipper, 3ファイル)
US-004	T-004	文書確認画面レイアウト変更 (3 controllers, 4ファイル)
```
